基础 自回归生成与 KV Cache

大模型一次只吐一个 token，每吐一个都要"回看"前面所有 token。KV Cache 把回看用的笔记存起来，避免重算。

点击播放：看 token 一个个生成，KV Cache 一格格填满。
输入 token（prefill） 新生成 token（decoding） 已存的 KV Cache
🔑 记住这一句：一个 token 的 K、V 只取决于它前面的 token。所以只要开头相同，这段 KV Cache 就能被别的请求直接拿去用 —— 这是后面所有优化的地基。

基础 共享前缀 → 复用 KV Cache

两个请求如果开头（前缀）相同，第二个可以跳过前缀的重复计算，直接复用第一个算好的缓存。

请求 B 来了，它的前缀和 A 一样……

核心1 RadixAttention：基数树管理缓存

SGLang 不再"用完即弃"，而是把所有请求的 KV Cache 放进一棵基数树，用 LRU 策略复用与淘汰。跟着步骤看这棵树怎么长、怎么分裂、怎么被淘汰。

新增节点 本步被复用（命中） 被淘汰（LRU） 已缓存

核心2 压缩 FSM：一步解码多个 token

约束输出成 JSON 时，像 {"summary": " 这样的固定串其实只有唯一走法。普通做法一个 token 一步（慢），压缩 FSM 把整段一步吐出。

普通 FSM：逐 token 解码

压缩 FSM：确定段一步跳过

左边每个 token 都要一次前向；右边固定串一次搞定，只在真正需要"生成"的地方停下。

核心3 API 推测执行：省一次调用

黑盒 API 模型改不了内部。SGLang 在第一次调用时"多生成几个 token"，赌它能顺手把后面的字段也写对，从而省掉一次调用和重复的输入费用。

朴素做法：2 次调用

推测执行：1 次调用

context 的输入 token，朴素做法要付两遍；推测执行只付一遍。

应用 自洽：共享前缀 + N 路采样 + 投票

同一问题独立跑 N 次（temperature>0 制造多样性），抽取并归一化各自答案，最后众数投票降低随机误差。共享的问题前缀在 SGLang 里只算一次。

同一道题，用 fork 分叉成 5 路，共享前缀的 KV Cache 只算一次。
共享问题前缀（缓存一次） 各自的推理链（随机采样） 投票胜出 少数派答案
🧮 投票背后的知识：① 统计/集成——多路边缘化、众数投票降方差；② 采样解码——靠 temperature>0 制造多样性；③ 答案抽取与归一化（0.5 = 1/2 才算同一票）；④ 投票规则——众数 / 处理平票 / 可加权。仅适用于答案离散可比的题。
