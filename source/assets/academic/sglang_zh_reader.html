LM Program：由代码组织的一组 LLM 调用，包含控制流、变量、并行和结构化输入输出。

摘要释义

一句话：SGLang 把复杂 LLM 应用看成“语言模型程序”，用 Python 内嵌的前端语言表达多次生成、并行、结构化输入输出，再用后端运行时复用 KV Cache、加速约束解码，从而提升吞吐和降低延迟。
新手提示 读摘要时先不要被系统名和优化名吓到。把它想成两层：上层让你更好写 LLM 工作流，下层让同样的工作流跑得更快。

论文关注的场景不是单轮聊天，而是代理、检索增强、多轮对话、少样本评测、JSON 生成、多模态推理等需要多次模型调用的工作流。作者认为，现有系统要么让程序员手动拼接字符串、解析输出和管理并行，要么把每次调用当成孤立请求执行，浪费了大量可以共享的前缀计算。

SGLang 由两部分组成：前端提供生成、选择、分叉、合并、图像/视频输入等语言原语；后端运行时提供 RadixAttention、压缩 FSM 和 API 推测执行等优化。论文报告在多类模型和任务上，相比当时常见系统可获得最高约 6.4 倍吞吐提升。

1 引言

LLM 的用法正在从“给一句提示，拿一次回答”变成“由程序安排一串模型调用”。典型例子包括 ReAct、Tree-of-Thought、自洽采样、少样本学习、工具调用、多模态代理和多轮聊天。它们都有共同特征：调用之间存在控制流和数据依赖，输入输出也常常需要结构化，以便继续交给后续软件组件处理。

为什么这篇论文重要 很多 LLM 应用的瓶颈已经不只是“模型会不会回答”，而是“几十次相关调用怎样组织、复用、并行和约束”。SGLang 把这个问题从应用层拉到了系统层。

论文把这类工作流称为 LM Programs。它们难写，原因包括提示字符串拼接脆弱、输出解析麻烦、并行控制复杂、结构约束和多模态输入难统一；它们也难跑快，因为不同调用之间经常共享长前缀，但通用推理引擎通常会重复计算 KV Cache。

SGLang 的核心主张是：既然 LM Program 天然包含可见的多调用结构，系统就应该利用这个结构。前端语言帮助开发者清楚表达“生成什么、何时并行、如何约束输出”；运行时则把这些结构信号转化为缓存复用、约束解码和 API 调用优化。

读图 1 的方法 从左往右看：用户写 SGLang 程序，解释器把程序拆成语言原语，运行时负责真正调模型。重点是前端不只是“语法糖”，它会把程序结构暴露给后端优化器。
前端语言Python 内嵌 DSL，提供生成、选择、分叉、合并和多模态输入等原语。
运行时管理请求批处理、KV Cache、约束解码和调度策略。
共同设计前端能把程序结构提示给后端，让后端做传统黑盒 API 看不到的优化。

2 编程模型

这一节用“多维度作文评审”作为贯穿例子：程序先把图像和作文放入提示，判断作文是否与图像相关；如果相关，就把提示状态分叉成多个并行分支，从不同维度评审；之后合并判断、生成总结，并按正则约束输出 JSON。

把 SGLang 想成什么 它不是替代 Python，而是在 Python 里给“提示状态”加了一组操作。你仍然用普通 Python 写 if、循环、函数，只是在需要模型参与时调用 SGLang 原语。

SGLang 的关键不是发明新的通用编程语言，而是在 Python 中加入适合 LLM 程序的少量原语。gen 负责生成并把结果写入变量，select 让模型从候选项中选择，extend 或 += 追加提示文本，fork/join 管理并行分支，image/video 处理多模态输入。

执行上，解释器把每个提示状态当作异步流处理，提交原语后可以继续运行 Python 逻辑，只有读取生成结果时才同步等待。这类似把多个生成任务排入流中执行。论文也讨论了把程序追踪成计算图的编译模式，用于更静态的优化。

易混点 fork 不是复制模型参数，而是复制当前提示状态，让多个后续生成分支共享同一个历史前缀。这正是后面 KV Cache 复用能发挥作用的地方。

3 使用 RadixAttention 高效复用 KV Cache

LLM 推理中的 KV Cache 只取决于前缀 token。只要两个请求共享前缀，后一个请求理论上就不必重新计算这段前缀的注意力中间结果。LM Program 正好大量出现这种情况：少样本例子相同、多轮聊天历史相同、树搜索分支前缀相同、多个程序实例共享系统提示。

直觉版解释 如果很多请求都以同一段系统提示或示例开头，模型每次都重新“读一遍”这段文本就很浪费。RadixAttention 的目标是让模型记住已经读过的公共前缀。

RadixAttention 把这些前缀缓存组织成基数树。树边不只代表单个 token，也可以代表一串 token，因此比普通 trie 更紧凑。每个节点关联对应的 KV Cache 页；请求到来时，运行时在树中匹配最长共享前缀，复用命中的缓存，再为新增部分插入节点。

缓存空间有限时，系统采用类似 LRU 的叶节点优先淘汰策略，并用引用计数避免移除正在被当前批次使用的节点。调度器还会倾向于选择能命中更多缓存的等待请求，从而提高批处理吞吐，但论文也承认公平性和饥饿问题还需要继续研究。

读这一节抓三个词 匹配：找到最长公共前缀；插入：把新出现的前缀路径写进树；淘汰：显存不够时优先丢掉最近少用、且无人使用的叶子节点。

4 使用压缩有限状态机加速约束解码

许多 LM Program 希望模型输出 JSON、固定字段、枚举值或其他可解析格式。SGLang 允许用正则表达式描述输出约束，再把正则转换为有限状态机，在解码时屏蔽非法 token。

为什么结构化输出会慢 约束解码不仅要问模型“下一个 token 概率是多少”，还要检查“这个 token 是否会违反格式”。如果每次只前进一步，固定格式越长，额外开销越明显。

普通约束解码常常每轮只生成一个 token，即使接下来有一段字符实际上已经被格式约束唯一决定。压缩 FSM 的思路是把连续的单一转移边合并成更长的边：如果接下来一段文本没有选择分支，就可以“跳跃式”前进，一次 forward pass 解析出多个 token 的效果。

由于正则是在字符/字符串层面表达，而模型工作在 token 层面，系统还需要重新分词，确保跳跃生成出的字符串与模型 tokenizer 的合法切分对齐。附录进一步讨论了这种方法在概率分布上的潜在偏差。

易混点 压缩 FSM 不是让模型随便跳过思考，而是当格式本身已经唯一决定下一段文本时，运行时直接推进这些确定性字符，再把结果重新对齐到 token。

5 面向 API 端点的推测执行

前面两个优化需要控制推理运行时，因此主要面向开源权重模型。对 GPT-4、GPT-3.5 这类只能通过黑盒 API 调用的模型，SGLang 不能直接修改解码器，但仍可以利用程序结构减少调用次数和输入 token 成本。

API 推测执行的做法是：当程序预计后续还会继续生成相邻字段时，第一次调用可以不在原定停止点立刻截断，而是让模型多生成一点；解释器保存这些额外输出，等后续 gen 原语到来时尝试匹配并复用。如果模板稳定、提示设计得当，后续调用就能被省掉或变短。

类比理解 这像是让模型一次多填几格表单。后面程序真的需要这些格子时，如果已经填对了，就不用重新请求 API；如果没填对，再退回正常调用路径。

6 实验评估

6.1 设置

实验覆盖 Llama-2、Mixtral、LLaVA 图像/视频模型以及 OpenAI GPT-3.5 API。硬件主要是 AWS G5 实例上的 NVIDIA A10G GPU，也包括部分 A100 GPU。对比对象既有 LMQL、Guidance 这类编程系统，也有 vLLM、TGI、TensorRT-LLM 等推理引擎。

工作负载包括 MMLU、HellaSwag、ReAct 轨迹、生成式代理、Tree-of-Thought、Skeleton-of-Thought、LLM judge、JSON 解码、检索增强生成、多轮聊天、多模态评测和 API 字段抽取。指标主要是最大吞吐和单实例平均延迟。

怎么看实验设置 入门读者可以先问两个问题：这些任务是否真的有共享前缀？这些任务是否需要结构化输出或并行分支？答案越是肯定，SGLang 越可能占优。

6.2 端到端性能

论文报告，在开源权重模型上，SGLang 的吞吐最高提升约 6.4 倍，延迟最高降低约 3.7 倍。不同任务的加速来源不同：少样本基准来自共享示例前缀，多轮/代理任务来自共享历史和分支，JSON 来自压缩 FSM，多分支工作流来自前端并行。

在更大的 Mixtral-8x7B 和 Llama-70B 张量并行设置中，趋势与 7B 模型类似。多模态模型中，运行时可用图像哈希作为缓存键来复用同一图像对应的 token 前缀。论文还提到 SGLang 曾部署在 Chatbot Arena，用真实流量观察到较高缓存命中率。

不要只看最高倍数 最高 6.4 倍说明上限很高，但更重要的是理解加速条件：共享前缀越多、分支越多、格式约束越强，系统优化的空间通常越大。

6.3 消融研究

消融实验显示，KV Cache 命中率越高，首 token 延迟和总延迟越低，同时可承载更大 batch，从而提升吞吐。移除树结构、缓存感知调度、前端并行或前端提示都会削弱效果。论文还测到 RadixAttention 在没有复用机会的负载上管理开销很小，因此可以默认开启。

对 JSON 解码，压缩 FSM 能显著提高吞吐；同时，FSM 预处理应当在批次之间复用，否则重复构建状态机会抵消收益。

消融实验的读法 消融不是为了展示“数字更多”，而是回答“到底是哪一个组件在起作用”。这里的结论是：树形缓存、调度、前端并行和结构化解码都各自贡献了一部分收益。

7 相关工作

论文把相关工作分为两条线：一条是 KV Cache 复用和推理系统优化，另一条是 LLM 编程、代理和提示框架。作者认为 RadixAttention 的特色在于把 KV Cache 当作树形 LRU 缓存来管理，并结合多级共享、缓存感知调度、前端运行时协同和分布式场景。

在编程系统上，SGLang 与 Guidance、LMQL 更接近，因为它保留对提示和生成原语的低层控制；但它同时带有自有运行时，所以能把语言级结构直接转化为推理级优化。高层框架如 DSPy、LangChain 或 AutoGen 可以把 SGLang 当作更底层的执行后端。

定位提醒 不要把 SGLang 只理解成 LangChain 的替代品。它更像是“低层生成控制语言 + 推理运行时”的组合，可以被更高层框架调用。

8 未来方向与结论

未来工作包括支持更多输出模态，把 RadixAttention 扩展到 DRAM/磁盘等多级存储，支持语义层面的近似匹配，增加更高层原语，改进缓存感知调度的公平性，以及增强编译器优化。

论文结论是：复杂 LLM 应用需要同时解决“怎么写”和“怎么跑快”。SGLang 用一组贴近 LM Program 的语言原语降低开发负担，再通过 RadixAttention、压缩 FSM 和解释器并行等运行时技术提升性能。

带走的核心观点 当 LLM 应用变成程序，系统就不能只优化单次请求。真正的优化对象会变成一整段调用图、共享前缀、并行分支和输出约束。
