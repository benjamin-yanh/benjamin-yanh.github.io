SGLang 学习博客 · 阶段一：从"看完了"到"被问倒了"

2026-07-06 · 论文：SGLang: Efficient Execution of Structured Language Model Programs（arXiv 2312.07104v2）

阶段一做了三件事：通读主学习笔记建立框架 → 用一轮追问式问答检验理解 → 启动 10 道自测题。这篇博客记录问答中沉淀下来的知识点、踩过的坑、以及暴露出的知识缺口。

1. 论文三十秒速览

SGLang 是一套让"LLM 程序"（多次模型调用 + 控制流 + 结构化输入输出）跑得更快的系统：前端是嵌入 Python 的 DSL（gen / select / fork / join 等原语），后端运行时带三项优化——RadixAttention（基数树 + LRU 自动复用 KV Cache）、压缩有限状态机（约束解码一次吐出多个 token）、API 推测执行（黑盒 API 模型省调用）。吞吐最高 6.4×，延迟最低降到 1/3.7。详细逐节讲解见主笔记，本文只写问答中真正"打通"的部分。

2. 问答沉淀的知识点

2.1 FSM 只会"过滤"，不会"生成"

约束解码的机制：模型每步照常做一次完整前向传播，输出全词表的分数；FSM 在这之后出场，查当前状态，把不合法候选的概率清零。FSM 是"事后审查员"，生成的重活儿仍全由模型做。

问题因此而来：格式里的固定串（如 {"summary": "）横跨多个 token，FSM 每一步都只剩唯一合法转移——答案注定，却仍要为每个 token 花一次昂贵的前向传播，纯粹走过场。压缩 FSM 的解法：把无分叉的"单行道"链合并成一条边，解码走到这里直接把整段 token 一次前向追加（类似 prefill），到真正有分叉、需要模型做选择处再恢复逐 token。

2.2 "唯一出口"从哪来：零概率、零数据

FSM 的唯一出口不是概率沉淀、也不是历史数据——它是从开发者写的正则当场编译出来的纯规则（regex → FSM 是经典的确定性算法）。正则在那个位置只写了一个字符，编译出的状态就只有一条出边。同一个正则编译一万次，结果一模一样。

类比：FSM 是道路设计图——哪里单行、哪里岔口，图纸画好就定死，与历史车流无关；模型是司机，只在岔口才需要做决定。概率全部住在模型的前向传播里。

2.3 一个反证法：约束解码为什么"永远合法"

命题：凡经 FSM 约束解码产生的完整输出 S，必匹配正则 R。

前提：① 自动机定理——S 匹配 R ⟺ 沿 S 在 FSM 上逐字符走，每步转移合法且终于接受状态；② 解码机制——每步先清零"走不通"的 token 才让模型选，且 EOS 只在接受状态合法。

反证：假设某次输出 S 不匹配 R，则只有两种可能——

情况 A：S 中某字符走了非法转移。取第一个出轨字符，它落在某个被采样的 token w 里。但 w 被采样前已被逐字符检查过；w 含非法转移 ⟹ 概率已被清成 0 ⟹ 不可能被采样。矛盾。
情况 B：每步合法但结束于非接受状态（如 JSON 缺右括号就停了）。但非接受状态下 EOS 概率为 0，解码停不下来。矛盾。

∎ 担保不来自模型、不来自数据，只来自"FSM 编译得对 + 清零逻辑实现得对"这两段确定性代码。

但它没保证什么？FSM 只懂语法，不懂语义。"summary": " 之后模型填一句胡话，逐字符全合法，FSM 全程放行。一句话：格式靠 FSM（确定性、可证明），内容靠模型（概率、不可证明）。

2.4 logits：概率的"毛坯"

一次前向传播的原始输出：给词表每个 token 打的一串生分数。流程：前向 → logits → softmax 归一化成概率 → 采样。约束解码插手的位置就在 logits 与 softmax 之间——把非法 token 的分数改成 −∞，softmax 后概率自然为 0。select 原语也是靠读 logits 比较各选项概率实现的。

2.5 开源 vs 闭源：谁掌握引擎，谁能优化

优化需要什么权限开源/自部署闭源 API
RadixAttention掌管 KV Cache（引擎内部）✓✗
压缩 FSM拿到 logits 做清零✓✗
API 推测执行只需能调 API✓（用不着）✓

闭源 API 拿不到 logits、管不了 KV Cache，所以三招里最值钱的两招上不了场——这就是论文单独为 API 模型设计第三招的原因。但"闭源"是相对于调用者的：服务商内部模型跑在自家引擎上，什么都摸得到。OpenAI 后来的 Prompt Caching 本质就是 RadixAttention 解决的同一件事，Structured Outputs 本质就是约束解码。结论：这些优化属于"谁掌握推理引擎，谁就能用"。

2.6 缓存命中：确定性内容 vs 随机性内容

"输入比输出更容易命中缓存"这个直觉要拆两半看：跨请求之间，命中几乎全靠输入（系统提示、few-shot 模板天然逐字节相同），输出是采样的、难复现；但同一对话内部，上一轮的模型回答成为下一轮前缀，输出的 KV 照样大量复用（多轮对话、fork 都是如此）。

KV Cache 只认"前缀是否逐 token 相同"，不区分这段 token 当初是输入还是输出。更精确的表述：确定性内容比随机性内容更容易命中——恰好输入多为确定性、新输出多为随机性。

2.7 两层 K/V：注意力的"货" vs 基数树的"账本"

RadixAttention 这个名字里藏着两套恰好撞名的 key/value：

keyvalue
注意力（模型计算层）每个 token 的"名片"向量——供后续 token 拿 Q 来匹配"该重点看谁"每个 token 的"内容"向量——匹配上后真正被取走混合的信息
基数树（缓存管理层）token 序列本身（沿边走的查找路径，纯整数数组）指针，指向这段序列的 K/V 张量所在的显存块

类比图书馆：注意力的 K = 书脊标签（供检索），V = 书的内容（检索命中后拿走的），Q = 检索词。基数树是账本，注意力 K、V 是账本里登记的货；比对树的 key 几乎免费，造货（矩阵乘法算 K、V）才贵，所以复用才划算。

用实例走一遍。设请求 A = 你|是|一个|助手|。|请|解|这道|题，请求 B = …请|写|个|故事，共享前缀 6 个 token：

  1. A prefill 时，模型每层为每个 token 算一对 (K,V) 向量（32 层模型 ⟹ "助手"一个 token 就有 32 对），存入显存，树里记下路径与指针；
  2. B 进来，拿它的 token 序列沿树逐 token 比对（纯整数比较，极便宜），命中前 6 个；
  3. 顺指针取走这 6 个 token 的 K/V 张量，prefill 只需为 写|个|故事 3 个 token 算新账——计算量 9 → 3；
  4. 树在"请"之后分叉成两条边——这正是论文图 3 画的分叉。

3. 踩坑记录

这部分是本阶段最值钱的内容：三个亲手踩出来的坑，每个都以为懂了、一说就露馅。

坑 1：把压缩 FSM 的逻辑装进 KV Cache 的问题里

✗ 错误回答："共享前缀可复用，是因为下一个 token 只有唯一出口、可预测。"

✓ 症结："唯一出口"是第 4 节压缩 FSM 的故事（省固定串的前向次数）；KV Cache 复用跟"下一个 token 可不可预测"毫无关系——哪怕下一个 token 完全随机，前缀的 KV 照样复用。两个优化解决的是两个不同的浪费。

坑 2：以为缓存里存的是"结果"，复用 = 少调用模型

✗ 错误回答："前缀在缓存中有相同的存的情况下可以立即获得结果，不需要等大模型输出；c 之后不分叉，输入 c 就能直接拼上 a。"

✓ 症结：三重误会一次拆清——① 输入不需要"预测"，请求到达时输入完整已知；② cat/car/card 是基数树怎么存和查的数据结构示例，不是预测机制；③ 复用的是"对已知输入做过的计算"（K/V 张量），发生在 prefill 阶段，省的是重算不是生成——该生成多少 token，模型一次前向都不少。

坑 3：K = 基数树，V = 运算结果？

✗ 错误理解：把注意力的 K、V 对号入座成"K 是基数树、V 是运算结果张量"。

✓ 症结：重名陷阱。注意力的 K/V 是模型计算层的两种向量（名片/内容）；基数树的 key/value 是缓存管理层的查找结构（路径/指针）。见 §2.7 的对照表。这个坑暴露的是前置知识缺口，而不是论文本身没读懂。

三个坑的共同规律：把"管理/约束层"的机制误认成"计算/生成层"的机制。SGLang 全文其实就是两层——模型照常算，系统在外面精打细算。分不清哪层在干活，就会把树、FSM、缓存都想象成"会预测的东西"。

4. 自测题进度（1/10）

第 1 题（为什么共享前缀可复用 KV Cache？）——三轮迭代后通关。标准答案：

因为一个 token 的 K、V 只取决于它前面的前缀、与后面内容无关，所以前缀相同的两个请求，这段前缀的 K、V 张量数学上完全相等。第一个请求 prefill 时算出的 K、V 缓存下来，第二个请求经前缀匹配后直接复用，跳过这段 prefill 重算；生成阶段照常，调用次数不变。收益：prefill 变短 ⟹ 首 token 延迟降低。

第 2–10 题待答（题目见主笔记第 10 节），留到阶段二。

5. 暴露的知识缺口与补课计划

坑 3 暴露的缺口：注意力机制内部构造（Q/K/V 到底是什么）。主笔记 §0.1(B) 刻意浅化，只给了"每个 token 一对 K/V、只取决于前缀"的结论——读论文够用，追问即露馅。

补课顺序（半小时~一小时，按序看）：

  1. 3Blue1Brown《GPT 是什么？》（系列第 5 章）——零基础起点，建立"token → 向量 → 层层变换 → 预测下一个词"的整体画面；
  2. 3Blue1Brown《注意力机制》（第 6 章）——正主，动画演示 Q/K/V 如何互相作用（B 站有中字）；
  3. Jay Alammar《The Illustrated Transformer》（有中译）——最后读，把视频建立的直觉用静态图固化，当速查手册。

边界提醒：SGLang 论文只用到 K/V 的三个事实——①每 token 每层一对向量；②算它们贵（prefill 主要成本）；③只取决于前缀。矩阵怎么乘、多头怎么拼，超纲不急。
